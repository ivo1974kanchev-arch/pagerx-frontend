PageRX API Documentation

Integrate page quality analysis into your tools, dashboards, and workflows. Everything you need to start scoring and analyzing web pages programmatically.

Base URL: https://api.pagerx.dev

Quick Start

Analyze any public web page in seconds. All endpoints accept a url parameter in the JSON body and return structured analysis results.

# Quickest way to test — returns a heuristic score
curl -X POST https://api.pagerx.dev/api/score/instant \
  -H "Content-Type: application/json" \
  -d '{"url": "https://example.com"}'

Endpoints

Four REST endpoints give you progressively deeper analysis. Choose the one that fits your use case.

POST /api/score/instant

Description

Returns an instant heuristic page score. Evaluates the page based on structural signals, content presence, meta tags, and best practices. Fastest endpoint — typically completes in under a second.

Request Body

{
  "url": "https://example.com"
}

Response

  "score": 82,
  "categories": {
    "seo":         { "score": 88, "label": "Good" },
    "accessibility": { "score": 76, "label": "Fair" },
    "performance":  { "score": 91, "label": "Excellent" },
    "content":     { "score": 74, "label": "Fair" }
  },
  "breakdown": {
    "has_title":       true,
    "has_meta_desc":   true,
    "has_h1":          true,
    "has_og_tags":     false,
    "load_time_ms":    340,
    "word_count":      1240,
    "img_alt_pct":     67,
    "internal_links":  12,
    "external_links":  4,

200 — Success 400 — Missing or invalid URL 422 — URL unreachable or invalid 500 — Server error

Code Examples

Integrate PageRX into your application with these ready-to-use snippets. All examples use the /api/analyze endpoint.

#!/bin/bash
# Analyze a page with PageRX using cURL

URL="https://example.com"

curl -X POST https://api.pagerx.dev/api/analyze \
  -H "Content-Type: application/json" \
  -d "{\"url\": \"$URL\"}" \
  | jq '.'
#!/usr/bin/env python3
"""PageRX API integration example."""

import requests
import json

def analyze_page(url: str) -> dict:
    """Analyze a web page using PageRX API."""
    response = requests.post(
        "https://api.pagerx.dev/api/analyze",
        json={"url": url},
        timeout=30
    )
    response.raise_for_status()
    return response.json()


# Example usage
if __name__ == "__main__":
    result = analyze_page("https://example.com")
    print(json.dumps(result, indent=2))
    print(f"Overall score: {result['overall_score']}/100")
// PageRX API integration — JavaScript / Node.js

async function analyzePage(url) {
  const response = await fetch("https://api.pagerx.dev/api/analyze", {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify({ url }),
  });

  if (!response.ok) {
    throw new Error(`HTTP ${response.status}`);
  }

  return await response.json();
}

// Example usage
analyzePage("https://example.com")
  .then(data => {
    console.log("Score:", data.overall_score);
    console.log("SEO issues:", data.perspectives.seo.issues.length);
  })
  .catch(err => console.error("Analysis failed:", err));

Batch Analysis (Python)

import requests
from concurrent.futures import ThreadPoolExecutor, as_completed

urls = [
    "https://example.com",
    "https://example.org",
    "https://example.net",
]

def score(url):
    r = requests.post(
        "https://api.pagerx.dev/api/score/instant",
        json={"url": url}, timeout=15
    )
    return url, r.json()["score"]

with ThreadPoolExecutor(max_workers=5) as pool:
    futures = {pool.submit(score, u): u for u in urls}
    for future in as_completed(futures):
        url, score = future.result()
        print(f"{url}: {score}/100")

Frequently Asked Questions

What types of pages can PageRX analyze? +

PageRX can analyze any publicly accessible web page, including landing pages, blog posts, e-commerce product pages, SaaS sites, and documentation pages. The page must be accessible over HTTP or HTTPS without authentication. We do not analyze pages behind login walls, paywalls, or private networks.

How is the score calculated? +

The heuristic score (/api/score/instant) is calculated from structural signals: presence of title tags, meta descriptions, heading hierarchy, image alt attributes, page load time, word count, link structure, and Open Graph tags. The full analysis (/api/analyze) adds perspective-specific checks like color contrast (accessibility), HTTPS enforcement (security), and content-to-HTML ratio (content quality). The LLM analysis uses a language model trained on web quality best practices.

Can I use PageRX in my CI/CD pipeline? +

Absolutely. PageRX is designed for automation. Use the /api/score/instant endpoint as a lightweight check in your CI pipeline (GitHub Actions, GitLab CI, Jenkins, etc.). The batch analysis pattern shown in the code examples section works well for automated quality gates. You can fail a build if the score drops below a threshold or if critical accessibility errors are found.

What happens when I exceed my rate limit? +

The API returns a 429 Too Many Requests status code with a Retry-After header indicating how long to wait before retrying. Free tier limits reset at the beginning of each calendar month. You can monitor your usage via the X-RateLimit-Remaining response header on every API call. Upgrade to Pro or Agency for higher limits.

Is there a sandbox or test environment? +

Yes! You can use https://example.com as a test URL — it's a stable, publicly available page that all endpoints can analyze. For more thorough testing, we recommend setting up your own test page or pointing the API at a staging environment. All endpoints return consistent, deterministic results for the same URL, making them suitable for integration tests.

How does the LLM analysis differ from the regular analysis? +

The standard analysis (/api/analyze) uses deterministic rules and heuristics — it's fast, consistent, and predictable. The LLM analysis (/api/analyze/llm) uses a language model to understand content nuance: writing quality, tone, brand voice, argument strength, and semantic structure. It provides human-like feedback but is slower and counts toward a separate usage limit. For automated quality gates, use the standard analysis; for deep content audits, use the LLM endpoint.
